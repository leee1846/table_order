Akamai WAF 환경에서 웹 표준인 SSE(Server-Sent Events) 통신이 자주 끊기고 정상 작동하기 어려운 핵심 원인을 4가지 논리로 간결하게 요약해 드립니다.

1. 프록시 버퍼링 아키텍처와 실시간성의 충돌 (표준 규격 훼손)
원인: Akamai 에지 서버는 보안 검사, 압축, 캐싱 효율을 위해 오리진에서 수신되는 응답 데이터를 에지 버퍼에 임시 적재한 후 전송하는 '버퍼링'을 기본 동작으로 수행합니다.

결과: SSE의 핵심 웹 표준인 청크 전송 인코딩(Transfer-Encoding: chunked) 헤더가 중간에서 박탈되고 정적인 Content-Length가 강제로 주입됩니다. 이로 인해 데이터가 실시간으로 흐르지 못하고, 에지 버퍼가 가득 차는 시점에 한꺼번에 쏟아져 나오는 '일시 배포(Batching/Bursting)' 현상이 발생하여 실시간 통신의 정체성이 무력화됩니다.

2. WAF 보안 검사와 스트리밍의 모순 (양자택일 딜레마)
원인: Akamai WAF(App & API Protector)가 SQL 인젝션이나 비정상 데이터 패킷을 실시간 감시하려면 전체 페이로드 문맥을 파싱해야 하므로 버퍼링이 구조적으로 필수적입니다.

딜레마: Akamai가 제공하는 '텍스트 스트리밍 최적화(Optimize Text Streaming)' 설정은 비즈니스에 극단적인 성능 유실을 강제합니다.

전송 최적화(버퍼링 Off): SSE 스트리밍은 정상 작동하지만, WAF의 심층 분석 기능(API Security, AAP 등)이 마비되어 보안 사각지대가 발생합니다.

보안 최적화(버퍼링 On): 보안은 철저히 유지되지만, 에지 버퍼링 장벽에 막혀 SSE 통신 자체가 불가능해집니다.

고가의 보안 장비를 도입하고도 보안을 끄거나 실시간 통신을 포기해야 하는 구조적 모순이 존재합니다.

3. 비대칭 타임아웃 구조로 인한 간헐적 끊김 (5xx 에러)
원인: Akamai 에지가 오리진 서버와 유지하는 지속 연결(PCONN)의 기본 유휴 타임아웃은 300초입니다.

결과: 백엔드 오리진 서버의 자체 Keepalive 타임아웃이 이보다 짧게 설정되어 있으면 오리진이 세션을 먼저 종료하게 되며, 이 과정에서 세션 정리 타이밍과 요청 중계 타이밍이 겹쳐 경쟁 상태(Race Condition)가 유발됩니다. 이로 인해 사용자에게 주기적으로 원인 불명의 5xx 게이트웨이 유실 오류 및 소켓 해제가 발생합니다.

동시성 제한: DDoS 방어를 위해 단일 사용자 IP당 허용하는 동시 연결 수(기본 50개)를 넘어서면 558 Connection Limit Stop 에러를 반환하며 연결을 강제 분쇄합니다.

4. 자가 설정 불가 및 변칙적 우회 (운영 오버헤드)
원인: Akamai 포털 UI에서는 에지의 내부 네트워크 계층이나 버퍼링 제어 관련 상위 파라미터를 사용자가 직접 수정할 수 없도록 제한하고 있습니다.

결과: 이를 해결하려면 반드시 Akamai 기술 지원 팀에 티켓을 열고 수주일에 걸쳐 수동 XML 메타데이터 설정을 반영해야 하므로 유연한 인프라 대응이 불가능합니다. 개발진은 이를 해결하기 위해 매 스트림 헤더마다 무의미한 8KB 이상의 공백 문자를 강제로 주입해 버퍼를 터뜨리는 비표준적 편법(Whitespace Padding Trick)을 소스코드에 매립해야 하는 치명적인 아키텍처 부채를 안게 됩니다.

결론 및 제언
SSE는 지속적인 단방향 스트림 연결을 위해 설계된 표준 프로토콜이지만 , Akamai WAF는 "데이터 패킷을 안전하게 취합 및 검사한 뒤 전송한다"는 상반된 철학을 가지고 있어 논리적으로 조화되기 어렵습니다.
